>>redis基础知识
1, redis是存储于内存中的高性能键值对的数据库;
2, 性能高: 内存+单线程+IO多路复用;
3, 丰富数据类型: 5种基础+5种扩展类型;
4, 持久化支持: RDB快照+AOF日志;
5, 高可用性: 主从复制+哨兵+集群;
6, 原子性操作: 单线程执行命令, 无并发竞争问题;
7, 功能丰富: 发布订阅、事务、Lua脚本、模块扩展等;

>>redis的使用场景
1, 缓存: 热点数据放redis, 设置过期时间 => 减轻数据库压力90%+;
2, 会话存储: 用户登陆状态存redis => 分布式系统共享Session;
3, 排行榜: Sorted Set 按分数排序 => 实时更新, 毫秒级查询;
4, 计数器: String类型的INCR/DECR => 原子操作, 高并发准确;
5, 分布式锁: SET key value NX EX 10 => 防止重复执行、超卖;
6, 消息队列: List的LPUSH/BRPOP => 简单异步任务处理;
7, 限流防刷: INCR+EXPIRE滑动窗口 => 控制接口访问频率;
8, 实时统计: HyperLogLog、Bitmap => 海量数据快速统计;
9, 地理位置: Geo类型 => 附近的人、距离计算;


>>安装后修改redis的配置文件 /etc/redis/redis.conf
1, daemonize no 改为 yes 
2, protected-mode yes 改为 no
3, bind 127.0.0.1 注释掉
4, requirepass 后添加密码


>>redis常用配置
daemonize yes  //让redis以守护进程的方式在后台运行, 默认为no
port 6379  //默认端口号
logfile ""  //日志文件, 保存redis底层的输出信息; 例如设置成 "redis.log", 那么原来输出到cmd中的信息就会保存到redis.log中;
databases 16  //默认容量16个数据库(0-15), redis数据库的名称就是0-15中的数字, 通过 select n 来切换数据库;
requirepass 12345  //设置redis的密码, 默认该项禁止; 在客户端通过密码认证: auth 12345;
dir ./   //数据文件的保存目录, 数据持久化, dump.rdb 就是保存在硬盘的备份文件;
protected-mode yes  //保护模式, 默认为yes, 如果其他主机需要连接redis, 需要修改该参数为no;
bind 127.0.0.1  //可以访问redis的主机ip, 默认为仅本机可访问, 修改为 0.0.0.0 表示所有主机都可访问;

>>redis的五种基本数据类型
0, 数据库以及对键的操作
select 0  //选择0号数据库
flushdb   /清空当前库
flushall  //通杀所有库
move key n  //当当前库的key移动到n号库
keys he*  //查询符合条件的key, 使用 keys * 查看所有的key
dbsize  //返回当前库中key的个数
help @类型  //查询指定类型的所有操作命令
1, String
set key value [nx | xx] [get] [ex 秒数 | px 毫秒数] [keepttl]
~ nx: 键不存在时设置  xx 键存在时设置
~ get: 返回设置前的值
~ ex 秒数 | px 毫秒数: 设置过期时间 
~ keepttl: 保留设置前键的生存时间
get name  //获取 name 的值
setnx name zhao   //不存在时则设置值, 可用于分布式锁
setex name 20 zhao  //设置值的同时设置过期时间 
getset name zhao  //先get再set
strlen name  //name的值的长度
mset name zhao age 30  //一次设置多个值
mget name age  //一次获取多个值
expire hello 20  //设置 key=hello 20秒后过期
ttl hello  //查看 key=hello 的过期剩余时间
exists a  //返回 key=a 是否存在
del a  //删除 key=a 的数据
getrange name 1 4   //截取字符串: abcd1234 => bcd1 
setrange name 1 ooo  //设置子字符串: abcd1234 => aooo1234
append name ooo  //在字符串后追加
incr/decr count  //count值自增/自减
incrby/decrby count 10  //自增/自减指定步长
setbit k1 2 1  //00100000  //strlen k1 : 1
setbit k1 9 1  //00100000 01000000  //strlen k1 : 2
注: 这是bitmap类型, 占用空间小, 实际开发中有时非常有用;
bitmap的应用场景:
1)统计某个用户在指定窗口时间内的登录天数;
2)统计某段时间有多少活跃用户;
2, Hash
用于存储结构化数据, 如 emp:1 -> {name: "zhao", age: 30, city: "hefei"};
与String不同, Hash的value为结构化数据;
hset emp:1 name zhao  //设置1号员工的name值
hget emp:1 name  //获取1号员工的name值
hmset emp:1 name zhao age 30  //一次设置1号员工的多个值 
hmget emp:1 name age / hgetall emp:1  //一次获取1号员工的多个值 / 获取1号员工的所有值
hdel emp:1 age  //删除1号员工的age
hexists emp:1 age  //返回1号员工的age是否存在
hlen emp:1  //返回1号员工有多少个属性值
hkeys emp:1  //返回1号员工所有的键
hvals emp:1  //返回1号员工所有的值
hgetall emp:1  //返回所有的键和值
hincrby/hincrbyfloat emp:1 age 10  //给指定属性增加一个数
hsetnx emp:1 age 20  //只有当指定属性不存在时才赋值
3, List
value的值为列表, 如 mylist -> {d, e, f, c, b, a};
rpush mylist c b a  //右侧插入
lpush mylist f e d  //左侧插入
rpop mylist  //右侧弹出
lpop mylist  //左侧弹出
lrange mylist 0 -1  //输出列表中所有元素
llen mylist  //列表中元素个数
lindex mylist 2  //获得指定下标的元素
lset key index value  //设置指定下标的值
linsert mylist before/after zhao qian  //在指定的值前或后插入值
lrem mylist n value  //删除n个指定value, n为0表示删除所有
ltrim key start end  //截取key中子字符串重新赋值给key
rpoplpush source dest  //从dest中右pop一个元素左push到source中
注: 应用举例
微信公众号消息订阅;
4, Set
value的值为集合, 与List不同, 集合的元素无序且唯一;
sadd s1 a b a c  //给集合添加元素
smembers s1 //查看集合中所有元素
sismember s1 a  //判断元素是否在集合中
scard s1  //获取集合里的元素个数
srem s1 a  //删除集合中的元素
srandmember s1 [n]  //从集合中随机展现n个元素(省略n表示1个), 不删除
spop s1 [n] //从集合中随机弹出n个元素, 删除
sinter s1 s2  //交集
sunion s1 s2  //并集
sdiff s1 s2  //差集
smove s1 s2 a  //将s1中的a元素移到s2中
sintercard 2 s1 s2  //返回两个集合s1和s2交集的元素个数
注: 应用举例 
抽奖程序; 朋友圈查看同赞的好友; 推荐可能认识的人;
5, Zset(Sort Set, 有序集合)
value的值为有序集合, 其中的元素由权和值来表示;
zadd z1 20 a 10 b 30 c  //给有序集合添加元素
zcard z1  //获取有序集合的元素个数
zrange z1 0 -1 [withscores]  //查看有序集合中所有元素, 可选的withscores表示显示分数
zrevrange z1 0 -1  //逆序显示 
zrangebyscore z1 10 20  //查看指定分数范围内的元素
zrevrangebyscore z1 20 10  //从大到小查看
zcount z1 60 90  //获取指定分数范围内的元素个数
zscore z1 a  //获取指定元素的分数
zrem z1 a  //删除指定元素
zincrby z1 10 a  //将a元素的分数加10
zrank z1 a  //获取按分数排名(从小到大)a的名次
zrevrank z1 a  //获取按分数排名(从大到小)a的名次
注: 应用举例
根据销售额对商品排序;


>>Bitmap类型(位图)
1, 用二进制位来保存0和1的数据, 以字节为单位来分配长度, 底层是String类型;
常用于统计用户登陆、签到, 广告点击等场景;
2, 常用操作
1)setbit bm n 1  //将第n位设为1, 此时strlen(bm)为1
2)getbit bm n    //取第n位的值
3)bitcount bm [m n] [byte|bit]  //返回[m, n]个byte或bit中1的数量, 默认是byte 
bitcount bm 表示返回bm中1的个数
4)bitop and|or|not|xor bm bm1 bm2  //对bm1和bm2进行位运算, 结果保存到bm
3, 应用场景
1)记录用户每月的签到
setbit uid:1001:202601 2 1
setbit uid:1001:202601 30 1
bitcount uid:1001:202601  //统计该用户本月签到的天数
2)记录某天有哪些用户登陆
hmset uid:map 0 1001 1 1002 2 1003  //将用户和位置的映射保存到一个hash中
setbit 20260101 1 1 
setbit 20260101 5 1
bitcount 20260101  //统计该天有多少用户登陆
bitop and bm 20260101 20260102  //统计0101和0102这两天连续登陆的用户数
bitcount bm  


>>HyperLogLog类型(基数统计)
1, 是一种去重统计功能的基数估计算法, 有以下特征: 
1)类似于Set, 但Set保存具体元素, 而HyperLogLog不保存具体元素, 只计算基数;
2)用于元素的个数非常大的情形, 该算法所需内存是固定的且很小, 12KB左右;
3)是估算, 不精确, 误差在0.81%左右;
4)底层也是String类型;
注: 基数是指一种数据集去重后的个数;
2, 常用操作
1)pfadd k a b c ...  //添加元素
2)pfcount k  //返回基数估算值
3)pfmerge k k1 k2 ...  //合并多个HyperLogLog
3, 应用场景
统计天猫网站首页亿级UV的方案(UV: unique visitor);


>>GEO类型(地理空间)
1, 一种能存储经纬度坐标的类型;
GEO的底层类型是Zset;
2, 常用操作 
1)geoadd cities 10 1 beijing 11 2 shanghai 12 3 hefei  //添加一个或多个位置的经纬度, 如果位置已存在则更新
2)geopos cities hefei  //返回指定位置的经纬度
3)geodist cities beijing hefei km|m  //计算两个位置的距离
4)geohash cities hefei  //返回指定位置的hash字符串
5)geosearch cities frommember beijing byradius 100 km   //以指定位置为中心, 按圆形搜索
geosearch cities frommember beijing bybox 500 300 km  //以指定位置为中心, 按矩形搜索
geosearch cities fromlonlat 116 39 byradius 100 km  //以指定经纬度为中心, 按圆形搜索
注: zset类型的基本操作也适用于GEO类型;


>>Stream类型(流)
1, redis版本的消息队列, 其底层就是stream类型, 而不是基于前面的基础类型;
Stream类型实现消息队列, 它支持消息的持久化、支持自动生成全局唯一的ID、支持ack确认消息的模式、
支持消费组模式, 让消息队列更加稳定和可靠;
2, redis实现消息队列的三种方案:
1)List方式, 只能用于点对点的模式, 不能用于广播, 常用来做异步队列;
2)pub/sub方式, 可用于广播, 但缺点是消息无法持久化, 而且无法保证数据的可靠性;
3)Stream类型, redis版的MQ消息中间件+阻塞队列, 克服了上面两种方式的缺点;
3, Stream的结构
Stream: 用一条链表存储所有消息, 每条消息有一条唯一ID, ID类似于主键;
消息ID: 格式为 时间戳-序列号, 可自动生成或手动指定;
消费组: 多个消费者建的群;
消费者: 消费组中的消费者;
last_delivered_id: 消费组所有, 游标, 表示消费者组已消费到的消息队列的位置;
pending_ids: 消费者所有, 消费者已读已确认的消息这叫ack, 已读未确认的消息放在 pending_ids[] 数组中;
4, 常用操作
//生产消息的命令
1)xadd   //添加消息到队列末尾, 消息ID必须比上个ID大, 可以用*来自动生成ID
xadd s * name zhao age 30  //返回自动生成的ID
xadd s 1774307538574-0 name zhao age 30  //手动指定ID
注: 一定要保证后面的ID比前面的大;
2)xlen s  //消息列表的长度
3)xrange s - + [count n]  //获取消息列表, -表示最小ID, +表示最大ID, 可选的n表示最多获取n条消息
4)xrevrange s + - [count n]  //按ID从大到小获取
5)xdel s id  //删除指定id的消息
6)xtrim  //截取消息队列
xtrim s maxlen n  //截取最后n条消息赋给sm
xtrim s minid id  //截取大于等于指定id的消息
7)xread [count n] [block m] streams s $|0   //获取消息(阻塞/非阻塞)
count n 最多读取n条, 不指定则表示所有
block m 是否以阻塞的方式读取, 默认不阻塞, m为0表示永远阻塞 
0|0-0表示大于最小ID
$表示大于最大ID, 配合阻塞方式读取下一条最新消息:
xread count 1 block 0 streams s $
注: 使用xadd存入消息和xread循环阻塞读取消息的方式可以实现简易版的消息队列, 同List;
//消费消息的命令
1)xgroup create  //创建消费组
xgroup create s groupA id|0|$  //0表示从stream的头部开始消费, $表示从尾部开始消费(即只消费最新消息)
注: 创建消费组时必须指定id, 通常设为0或$;
2)xreadgroup group  //消费组消费消息
xreadgroup group groupA A1 [count n] [block m] streams s >
'>' 表示从第一条尚未被消费的消息开始读取;
省略count则表示读取全部未读消息;
stream中的消息一旦被消费组里的一个消费者读取了, 就不能在被同组内的其他消费者读取;
不同消费组的消费者可以读取同一条消息;
3)xack  //消费者对读取消息的确认
xack s groupA id  //确认指定id的消息
4)xpending  //查询消费组
xpending s groupA  //查询消费组groupA所有消费者已读取但未确认的消息
xpending s groupA - + n A1  //查询消费组groupA的消费者A1已读取未确认的前n条消息
5)xinfo  //查询stream/group/consumer的详细信息
xinfo stream s 
xinfo groups s 
xinfo consumers s groupA


>>bitfield类型(位域)
1, 将一个redis字符串看成是一个由二进制位组成的数组, 将其划分为多个区域, 每个区域独立存储整数;
底层是String类型; 实际开发中使用较少; 



>>持久化: RDB & AOF 
1, rdb(默认)
1)把某一时刻的数据集快照(snapshot)以文件的形式写到磁盘上, 该文件为dump.rdb;
写入时机分为自动触发与手动触发两种方式;
2)自动触发
在指定时间间隔内修改指定的次数就会触发持久化操作:
save seconds changes
redis6: save 900 1 300 10 60 1000
redis7: save 3600 1 300 100 60 10000
注: save "" 表示禁用rdb;
3)手动触发
bgsave(默认) / save 
两者区别: 
save会阻塞当前redis服务器, 生产时禁止使用; 而bgsave会fork一个子进程来异步处理保存操作, 
不会阻塞服务器;
4)rdb常用配置选项
save 3600 1  //rdb持久化规则
dir /etc/redis/dumpfiles   //rdb文件保存路径
dbfilename dump.rdb   //rdb文件名
stop-writes-on-bgsave-error yes  //保存失败时则停止更新数据
rdbcompression yes  //对rdb文件进行压缩
rdbchecksum yes  //对rdb文件进行数据校验
5)rdb的优缺点
优点: 适合大规模的数据恢复、定时备份、rdb文件在内存中的加载速度要比aof文件快得多;
缺点: 宕机会丢失最近一次快照期间的数据;
6)修复rdb文件
redis-check-rdb /etc/redis/dumpfiles/dump.rdb
7)触发rdb快照的情形
自动配置、手动配置、
flushdb/flushall(此时dump.rdb为空)、
执行shutdown且没有开启aof、
主从复制时主节点自动触发;
2, aof
1)把对redis的每次写操作保存到appendonly.aof文件中, 该文件只追加, 不修改;
aof持久化功能默认不开启, 要在配置文件中设置 appendonly yes;
2)aof文件的三种写回策略
对redis的写操作先存放在内存的缓冲区, 何时将缓冲区的内容写到aof文件有以下三种策略:
appendfsync always: 同步写回, 不会丢数据, 但影响性能;
appendfsync everysec(默认):每秒写回, 折衷方案;
appendfsync no: 由操作系统决定, 宕机时会丢数据;
3)aof常用配置选项
appendonly yes   //开启aof持久化
appendfsync everysec  //写回策略
appendfilename "appendonly.aof"  //aof文件名
appenddirname "appendonlydir"  //aof文件存放目录, 该目录会拼接在设置rdb时的dir目录后面
redis6只有一个aof文件, 但在redis7中, aof文件包含3个:
appendonly.aof.1.base.rdb  //base, 表示基础aof, 该文件只有一个
appendonly.aof.1.incr.aof  //incr, 表示增量aof, 该文件可以有多个
appendonly.aof.manifest  //清单, 用于管理所有aof文件
4)aof的优缺点
优点: 默认的everysec写回策略保证最多只丢失一秒钟的写入数据;
aof文件太大时能够在后台自动重写;
如果不小心执行了清空命令, 可以在aof文件中手动删除该命令, 然后重新加载aof文件就可以找回数据;
缺点: aof文件比同级的rdb文件大, 恢复时的速度也比rdb慢;
5)aof文件异常修复
redis-check-aof --fix /etc/redis/dumpfiles/appendonlydir/appendonly.aof.1.incr.aof 
注: base和manifest文件不用修复, 只需要修复incr文件;
6)aof重写机制
分为自动重写和手动重写两种;
//自动重写
aof文件大小超过了配置文件中设定的峰值, 则会自动进行重写:
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
//手动重写
bgrewriteaof
3, rdb和aof混合持久化(推荐)
1)开启混合持久化(默认开启)
aof-use-rdb-preamble yes  
2)加载流程
开启混合持久化时, 只从aof文件中恢复数据; 只有当aof文件不存在时, 才加载rdb文件;
3)推荐使用混合持久化
rdb镜像做全量持久化, aof做增量持久化;
混合持久化产生的文件一部分时rdb格式, 一部分是aof格式;
4, 纯缓存模式
可以同时关闭rdb和aof的持久化功能, 将redis数据库作为纯高速缓存来用:
save ""  //禁用rdb
appendonly no  //禁用aof
